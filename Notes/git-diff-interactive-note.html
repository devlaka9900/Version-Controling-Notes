Git lesson note

Understanding git diff

A simple, detailed, interactive guide to comparing changes in files, commits, and branches. Think of it like comparing two drafts of the same novel: Git helps you see exactly what changed.

Jump to cheat sheet Start practice

1. What you will learn

Compare files

See exactly what lines were added, removed, or changed in your working files.

Compare commits

Check how your project changed between two saved points in history.

Compare branches

Understand what is different between branches such as main and a feature branch.

2. The big idea: git status vs git diff

git status tells you which files changed. But it does not show the actual content changes. git diff goes one level deeper and tells you what exactly changed inside those files.

Command Simple meaning Example result
git status Shows the names of files that changed. modified: README.md
git diff Shows the actual line-by-line changes. Shows removed lines with - and added lines with +.
Quick memory trick

Imagine your project is a folder of drafts. git status says, "Draft 3 changed." git diff opens Draft 3 and highlights the exact sentences that changed.

3. Why git diff matters

Great novels are rarely written in one try. Authors create drafts, remove ideas, bring back old sentences, and polish the final version. Programming is similar. You often edit code, remove lines, test ideas, and later need to check what changed.

Without a comparison tool, you may forget what you changed. git diff helps you avoid that confusion by showing the difference between two versions of your project.

4. Simple example: CITIES.txt

Suppose you have a file called CITIES.txt that stores cities you visited during a South America tour. Later, you edit the file but forget what you changed. git diff can compare the older version with the current version and show the exact differences.

Version A

Buenos Aires
Lima
Quito
Santiago

Version B

Buenos Aires
Lima
Bogota
Santiago

Git may show a diff like this:

diff --git a/CITIES.txt b/CITIES.txt
--- a/CITIES.txt
+++ b/CITIES.txt
@@ -1,4 +1,4 @@
 Buenos Aires
 Lima
-Quito
+Bogota
 Santiago
How to read this output
  • -Quito means Quito was removed.
  • +Bogota means Bogota was added.
  • Lines without + or - are unchanged context lines.

5. Anatomy of a git diff output

Part Meaning Easy explanation
diff --git a/file b/file Shows the two file versions being compared. a/ usually means old version. b/ usually means new version.
--- a/file Marks the old file. This is the version before your change.
+++ b/file Marks the new file. This is the version after your change.
@@ -1,4 +1,4 @@ Shows where the change happened. This is called a hunk header.
- old line Removed line. This line existed before, but not now.
+ new line Added line. This line exists now, but did not exist before.

6. Practice lab: compare a changed file

This practice follows the same idea from the lesson. You edit README.md, then use git diff to inspect the actual change before committing.

  1. Check your current branch and file state. 
    git status
  2. Edit a file.

    You can use VS Code, Vim, Nano, or any editor.

    vim README.md
  3. Check which file changed. 
    git status
  4. View exactly what changed compared with HEAD. 
    git diff HEAD -- README.md
Why use HEAD here?

HEAD points to the current commit you are working from. So git diff HEAD -- README.md asks Git: "Compare my current README.md with the version stored in the current commit."

7. Three common ways to use git diff

A. Compare unstaged changes

Shows changes in your working directory that are not staged yet.

git diff

B. Compare staged changes

Shows changes you already added using git add.

git diff --staged

C. Compare with HEAD

Shows all current changes compared with the latest commit.

git diff HEAD

D. Compare one file

Focus only on one file instead of the whole project.

git diff HEAD -- README.md

8. Compare two commits

A commit is a saved snapshot of your project. Each commit has a unique hash ID. To compare two commits, first view your commit history, then pass two commit IDs to git diff. 

git log --oneline

Example output:

9c90abc Add minor update
8b55def Create first README
3a21f00 Initial commit

Now compare the newest commit with an older commit:

git diff 3a21f00 9c90abc
Order matters

git diff old new shows what changed to move from old to new. If you reverse the order, added lines can appear as removed lines, and removed lines can appear as added lines.

9. Compare two branches

You can use git diff to see what is different between branches. This is useful before merging a feature branch into main or master.

  1. List the branches. 
    git branch
  2. Compare main with a feature branch. 
    git diff main feature/testing
  3. Compare only changes introduced by the feature branch.

    This uses the merge base and is often helpful before opening or reviewing a PR.

    git diff main...feature/testing
Two dots vs three dots

git diff main..feature/testing compares the two branch tips directly. git diff main...feature/testing compares from the point where the branches split, which is usually better for reviewing what the feature branch added.

10. Visual idea

Here is a simple way to picture branch comparison:

main:            A --- B --- C
                       \
feature/testing:         D --- E

Command:
git diff main...feature/testing

Meaning:
Show what changed in D and E compared with the place where feature/testing split from main.

11. Before you commit, review with this checklist

12. Git diff cheat sheet

Goal Command Use when
See unstaged changes git diff You edited files but did not run git add yet.
See staged changes git diff --staged You ran git add and want to review before committing.
Compare current work with latest commit git diff HEAD You want all current changes compared with your current commit.
Compare one file with latest commit git diff HEAD -- README.md You only care about one file.
Compare two commits git diff oldCommit newCommit You want to see what changed between two saved snapshots.
Compare two branches git diff main feature/testing You want to see differences between branch tips.
Review feature branch changes git diff main...feature/testing You want to see what the feature branch introduced since it split from main.
Show only changed file names git diff --name-only You want a quick list instead of full line changes.
Show summary of changes git diff --stat You want a compact summary of insertions and deletions.

13. Common mistakes and fixes

I ran git diff, but nothing appeared.

This can happen if there are no unstaged changes. Try git status. If your changes are already staged, use git diff --staged.

I want to compare only one file.

Use -- before the file path. Example: git diff HEAD -- README.md. The -- helps Git understand that what follows is a file path.

The output is very long.

Use git diff --stat for a summary or git diff --name-only to list only file names.

I want to exit the diff screen.

If Git opens the output in a pager such as less, press q to quit.

15. Final summary

git diff is your detail-view command. Use it when you want to understand exactly what changed before committing, merging, or reviewing a branch. It helps you avoid accidental changes, review your work clearly, and compare different versions of your project with confidence. 

git status          # Shows which files changed
git diff            # Shows unstaged line changes
git diff --staged   # Shows staged line changes
git diff HEAD       # Shows all current changes compared with HEAD
git diff A B        # Shows differences between commit or branch A and B
