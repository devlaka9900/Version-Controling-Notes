How to use this practice guide

Use one assignment per day when you are busy, or three assignments per session when you want a deeper practice block. For best memory retention, type the commands manually instead of copying them. Before opening an answer, write your own command sequence in a scratch file.

Daily rhythm

  1. Read the scenario.
  2. Try the task without the answer.
  3. Run git status often.
  4. Open the answer and compare.
  5. Repeat the command from memory once.

Practice rule

Always use a sandbox repository. Never practice destructive commands such as reset --hard or clean -fd inside important work unless you understand the result.

Best habit

Use git status, git diff, and git log --oneline --graph --decorate --all constantly. These commands tell you where you are before you change history.

One-time setup for the practice environment

Run these commands once in a safe folder. The assignments below assume your main practice repository is named daily-notes.

# Check that Git is installed
git --version
# Configure the name recorded in your commits
git config --global user.name "Your Name"
# Configure the email recorded in your commits
git config --global user.email "you@example.com"
# Use main as the default branch name for new repositories
git config --global init.defaultBranch main
# Create a parent folder for all Git practice repositories
mkdir git-practice-lab
# Move into the parent folder
cd git-practice-lab

Tip: If you already have a global Git config, you can still use this lab safely. These commands only set common beginner-friendly defaults.

The Git mental model you should memorize

Four important places

PlaceMeaningCommon commands
Working directoryYour actual files on disk.git status, git diff, git restore
Staging area / indexThe next snapshot you are preparing.git add, git restore --staged, git diff --staged
Local repositoryYour commit history stored in .git.git commit, git log, git show
Remote repositoryA shared copy, such as GitHub, GitLab, or a local bare repository.git fetch, git pull, git push

Simple memory map

# See what changed and where you are
git status
# Move selected file changes into the next snapshot
git add file.txt
# Save the staged snapshot to local history
git commit -m "type: clear message"
# Download remote history without merging
git fetch origin
# Merge remote changes into the current branch
git pull
# Upload local commits to the remote
git push

Daily self-check

Commands to respect

These commands are powerful. They are not bad, but they can discard or rewrite work when used carelessly.

CommandRiskSafer habit
git reset --hardDiscards uncommitted changes and moves the current branch.Run git status, git diff, and consider a temporary commit or stash first.
git clean -fdDeletes untracked files and folders.Preview first with git clean -nd.
git push --forceCan overwrite shared remote history.Prefer --force-with-lease only when team workflow allows rewriting.
git rebaseRewrites commit hashes for the rebased commits.Use freely on private local branches; coordinate before rebasing shared branches.

Level 1 - Beginner Foundations

Assignment 01

Create your first repository and first commit

Scenario: You started a new personal notes project and want Git to track it from day one.

Objectives

  • Initialize a repository
  • Create a file
  • Stage and commit the first snapshot
  • Inspect the commit history

Your tasks

  1. Create a folder named daily-notes.
  2. Initialize Git inside it.
  3. Create README.md with one sentence.
  4. Commit it with a clear message.
  5. View the log in one-line format.
Show answer and command sequence 
# Create a practice folder for the project
mkdir daily-notes
# Move into the project folder
cd daily-notes
# Start a new Git repository in the current folder
git init
# Create the first tracked file with simple content
echo "# Daily Notes" > README.md
# Show the current Git state before staging
git status
# Add README.md to the staging area
git add README.md
# Save the staged snapshot as a commit
git commit -m "docs: add project readme"
# Show compact commit history
git log --oneline

Success checks

  • git status should say the working tree is clean.
  • git log --oneline should show one commit.

Common pitfalls

  • Running git init in the wrong folder
  • Forgetting to stage before committing

Objectives

  • Use git diff --staged

Your tasks

  1. Edit README.md twice.
  2. Stage only after the first meaningful change.
  3. Compare unstaged and staged changes.
  4. Commit the staged version.
  5. Check what remains unstaged.
Show answer and command sequence 
# Add a first change to README.md
echo "Today I practiced Git basics." >> README.md
# See the unstaged modification
git status
# See exact unstaged line changes
git diff
# Stage the current README.md change
git add README.md
# Add another change after staging
echo "Tomorrow I will practice branches." >> README.md
# See staged and unstaged areas separately
git status
# See only unstaged changes
git diff
# See only staged changes

Success checks

  • git diff --staged should be empty after the commit.

Common pitfalls

  • Thinking git commit saves every file automatically
  • Not checking git diff before committing

Scenario: You found a confusing line and need to learn when and why it was introduced.

Objectives

  • Use git blame
  • Inspect responsible commit
  • Understand code archaeology

Your tasks

  1. Run blame on README.md or main.js.
  2. Copy a commit hash from the output.
  3. Show that commit.
  4. Summarize why the line changed.
Show answer and command sequence 
# Show the last commit responsible for each line in README.md
git blame README.md
# Inspect a specific blamed commit by replacing HASH with the commit hash
git show HASH
# Show only changed files for that commit
git show --name-status HASH
# Show surrounding history for README.md
git log --oneline -- README.md

Success checks

  • You should identify the commit hash responsible for a line.
  • You should understand the commit message and diff related to that line.

Common pitfalls

  • Using blame to blame people instead of understanding context
  • Ignoring surrounding commits and assuming one line tells the full story
Assignment 43

Handle detached HEAD and save work

Scenario: You checked out an old commit, made changes, and now Git says you are in detached HEAD state.

Objectives

  • Recognize detached HEAD
  • Create a branch to save work
  • Return to main safely

Your tasks

  1. Check out an older commit by hash.
  2. Create a change and commit it while detached.
  3. Save it by creating a branch.
  4. Switch back to main.
Show answer and command sequence 
# View recent commits and choose an older hash
git log --oneline -5
# Check out the older commit by replacing OLD_HASH with a real hash
git switch --detach OLD_HASH
# Confirm detached HEAD state
git status
# Make a commit while detached for practice
echo "Detached experiment" > detached.md
git add detached.md
git commit -m "test: add detached experiment"
# Save the detached commit by creating a branch at the current commit
git switch -c saved/detached-experiment
# Return to main safely
git switch main
# Confirm the saved branch exists
git branch --list saved/detached-experiment

Success checks

  • The detached commit should be reachable through saved/detached-experiment.
  • main should remain unchanged until you merge or cherry-pick.

Common pitfalls

  • Switching away from detached HEAD without saving useful commits
  • Panicking when detached HEAD is only an informational state
Assignment 44

Simulate a fork-style contribution with two remotes

Scenario: You want to understand upstream and origin like an open-source workflow.

Objectives

  • Use multiple remotes
  • Fetch upstream
  • Push to origin
  • Keep fork updated

Your tasks

  1. Create upstream.git as the main project remote.
  2. Clone it as my-fork-working-copy.
  3. Rename/add remotes to simulate origin and upstream.
  4. Fetch upstream and push to origin.
Show answer and command sequence 
# Move above your current project folder
cd ..
# Create a bare upstream repository
git init --bare upstream.git
# Clone upstream into a maintainer working copy
git clone upstream.git upstream-maintainer
# Enter the maintainer copy
cd upstream-maintainer
# Create initial upstream content
echo "Upstream project" > README.md
git add README.md
git commit -m "docs: initialize upstream project"
git push -u origin main
# Return above the folders
cd ..
# Create a bare fork repository by cloning upstream as bare
git clone --bare upstream.git my-fork.git
# Clone your fork as your working copy
git clone my-fork.git my-fork-working-copy
# Enter your fork working copy
cd my-fork-working-copy
# Add upstream remote pointing to the original project
git remote add upstream ../upstream.git
# List both remotes
git remote -v
# Fetch changes from upstream
git fetch upstream
# Create a contribution branch
git switch -c feature/open-source-practice
# Add contribution content
echo "Contribution idea" > contribution.md
git add contribution.md
git commit -m "docs: add contribution idea"
# Push your contribution branch to your fork origin
git push -u origin feature/open-source-practice

Success checks

  • git remote -v should show origin and upstream.
  • Your contribution branch should be pushed to origin, not upstream.

Common pitfalls

  • Pushing directly to upstream when you meant origin
  • Not fetching upstream before starting new work
Assignment 45

Accidentally committed a secret before pushing

Scenario: You committed a private-looking value locally, noticed immediately, and have not pushed yet.

Objectives

  • Remove secret from latest commit
  • Update .gitignore
  • Amend safely before push

Your tasks

  1. Create a fake local.secret file and commit it accidentally.
  2. Remove it from the Git index but keep or delete the local file.
  3. Add it to .gitignore.
  4. Amend the commit before pushing.
Show answer and command sequence 
# Create a fake secret file for practice only
echo "TOKEN=fake-practice-token" > local.secret
# Accidentally stage the fake secret
git add local.secret
# Accidentally commit the fake secret
git commit -m "chore: add local secret by mistake"
# Remove the file from Git tracking but keep it locally
git rm --cached local.secret
# Add the secret file pattern to .gitignore
echo "local.secret" >> .gitignore
# Stage the ignore rule and index removal
git add .gitignore
# Amend the latest commit so the secret is not in that commit
git commit --amend -m "chore: ignore local secret file"
# Confirm the secret file is not tracked
git status --ignored
# Confirm the latest commit does not include local.secret
git show --name-only --oneline HEAD

Success checks

  • local.secret should not appear in git show --name-only HEAD.
  • git status --ignored should show local.secret as ignored.

Common pitfalls

  • Assuming .gitignore removes files already committed
  • Using fake practice secrets in real services
Assignment 46

Accidentally pushed a secret: safe team response

Scenario: A secret-like value was pushed to a shared remote, so history rewriting alone is not enough operationally.

Objectives

  • Stop exposure
  • Rotate secret
  • Coordinate cleanup
  • Avoid unsafe assumptions

Your tasks

  1. Write the response plan as a checklist.
  2. Remove the secret from current code.
  3. Commit the removal.
  4. Coordinate credential rotation outside Git.
  5. Only rewrite history with team agreement and repository-owner tooling.
Show answer and command sequence 
# Remove the secret from the working tree or replace it with a safe placeholder
printf "TOKEN=read-from-environment\n" > local.secret.example
# Stop tracking the actual secret file if it is tracked
git rm --cached local.secret
# Add local secret file to ignore rules
echo "local.secret" >> .gitignore
# Stage safe files and ignore rules
git add .gitignore local.secret.example
# Commit the cleanup in shared history
git commit -m "fix: remove committed local secret reference"
# Push the cleanup commit to shared remote
git push
# Rotate or revoke the exposed credential in the real provider outside Git
# Coordinate with the team before any force-push or history rewrite
# Ask repository admins to run secret scanning and history cleanup if required

Success checks

  • The current branch should no longer contain the secret in normal files.
  • The real credential should be rotated or revoked outside Git.

Common pitfalls

  • Only deleting the file and thinking the exposed secret is safe
  • Force-pushing rewritten history without team coordination
Assignment 47

Create helpful Git aliases

Scenario: You forget long commands, so you want short aliases for daily inspection.

Objectives

  • Configure aliases
  • Use short commands
  • Keep aliases readable

Your tasks

  1. Create aliases for status, compact log, graph log, and staged diff.
  2. Use each alias.
  3. List Git config aliases.
Show answer and command sequence 
# Create a short alias for git status
git config --global alias.st status
# Create a compact one-line log alias
git config --global alias.lg "log --oneline --decorate"
# Create a graph history alias
git config --global alias.tree "log --oneline --graph --decorate --all"
# Create a staged diff alias
git config --global alias.ds "diff --staged"
# Use the status alias
git st
# Use the compact log alias
git lg
# Use the graph alias
git tree
# Use the staged diff alias
git ds
# List global aliases
git config --global --get-regexp '^alias\.'

Success checks

  • git st should behave like git status.
  • git tree should show a graph view.

Common pitfalls

  • Creating aliases so cryptic you forget them too
  • Overusing aliases before learning the full commands
Assignment 48

Practice a complete feature workflow

Scenario: You want a full end-to-end routine similar to professional team work.

Objectives

  • Sync main
  • Branch
  • Commit
  • Rebase
  • Merge
  • Push
  • Delete branch

Your tasks

  1. Update main from origin.
  2. Create a feature branch.
  3. Make two clean commits.
  4. Rebase on updated main.
  5. Merge into main.
  6. Push main and clean up.
Show answer and command sequence 
# Switch to main before starting work
git switch main
# Fetch latest remote state
git fetch origin
# Update local main from origin main
git pull --ff-only
# Create a focused feature branch
git switch -c feature/complete-workflow
# Create the first focused change
echo "Workflow step one" > workflow.md
git add workflow.md
git commit -m "docs: add workflow step one"
# Create the second focused change
echo "Workflow step two" >> workflow.md
git add workflow.md
git commit -m "docs: add workflow step two"
# Rebase feature branch onto the latest main
git fetch origin
git rebase main
# Return to main for integration
git switch main
# Merge the feature branch
git merge --no-ff feature/complete-workflow -m "merge: integrate complete workflow practice"
# Push updated main
git push origin main
# Delete local feature branch
git branch -d feature/complete-workflow

Success checks

  • main should contain workflow.md.
  • The feature branch should be deleted locally after merge.
  • git log --graph should show the integration clearly.

Common pitfalls

  • Starting from stale main
  • Skipping tests before merge
  • Deleting remote branches without confirming team workflow

Five-minute memory drills

Use these when you do not have time for a full assignment.

Drill A: inspect

# Check current branch and file state
git status
# Review unstaged changes
git diff
# Review compact history
git log --oneline -5

Drill B: commit

# Stage one selected file
git add file.txt
# Review what will be committed
git diff --staged
# Commit with a useful message
git commit -m "docs: describe the change"

Drill C: branch

# Create and switch to a branch
git switch -c feature/small-practice
# Return to main
git switch main
# Delete the branch if fully merged
