Version Control Systems

Detailed notes on how developers track code changes, collaborate safely, and manage project history using centralized and distributed version control systems.

1. What Is Version Control?

Version control is a system that helps developers track, manage, and store changes made to source code over time.

When developers work on software, the code is constantly changing. New features are added, bugs are fixed, files are updated, and old code may be removed. In a team environment, several developers may work on the same codebase at the same time.

Version control makes it easier to know who changed the code, what was changed, when it was changed, and why the change was made.

Without version control, teams may accidentally overwrite each other’s work, lose important code, or struggle to find the source of a bug. Version control solves these problems by keeping a history of changes.

2. Why Developers Use Version Control

Version control is important because software development is rarely done by one person in isolation. In real projects, developers usually collaborate in teams and work on different tasks at the same time.

  • One developer may be building a new feature.
  • Another developer may be fixing a bug.
  • Another developer may be improving performance.
  • Another developer may be updating documentation.

All these changes need to be organized carefully. Version control allows teams to combine changes, compare versions, recover older code, and understand the history of a project.

In simple words, version control acts like a memory system for your codebase.

3. Examples of Version Control Systems

There are many version control systems available. Some are older and centralized, while others are modern and distributed.

  • Subversion: A centralized version control system, often called SVN.
  • Perforce: A version control system often used in large enterprise or game development projects.
  • AWS CodeCommit: A source control service provided by Amazon Web Services.
  • Mercurial: A distributed version control system.
  • Git: A popular distributed version control system used by many developers.
Git is one of the most widely used version control systems today.

4. Two Main Types of Version Control Systems

Version control systems can be divided into two main categories:

Centralized Version Control System

A centralized system stores the main project history on a central server. Developers connect to that server to get code and submit changes.

Distributed Version Control System

A distributed system gives every developer a complete copy of the repository, including the full project history.

5. Centralized Version Control Systems

A Centralized Version Control System, also called CVCS, works using a server and client model.

The server contains the main repository. This repository stores the full history of the project. Developers use their computers as clients to connect to the server and access the latest code.

How CVCS Works

  • The main repository is stored on a central server.
  • Developers pull the latest version of the code from the server.
  • The developer gets a working copy on their local machine.
  • The local machine usually has the latest code, but not always the full history.
  • To view history, update code, or submit changes, the developer often needs to connect to the server.
  • After making changes, the developer pushes or commits those changes back to the central server.
In CVCS, the central server is the main source of truth for the project.

Simple Example

Imagine a team is using a centralized system. The project lives on one main server. Each developer downloads a working copy of the project. After making changes, they send those changes back to the same server.

Other developers must connect to the server to receive those updates.

Advantages of CVCS

CVCS can be easier for beginners because there is one main repository and one central place where the project history is stored.
  • Easier to understand: The workflow is usually simple because there is one central server.
  • Better centralized access control: Administrators can control who can access the server and what permissions each user has.
  • Clear central copy: Everyone knows that the main version of the project is stored on the central server.

Disadvantages of CVCS

CVCS depends heavily on the central server. If the server is slow or unavailable, developers may not be able to perform important actions.
  • Needs server connection: Many operations require access to the central server.
  • Can be slower: Since actions often need the server, performance can depend on network speed.
  • Limited offline work: Developers may not be able to view full history or perform some actions without being connected.
  • Single point of failure: If the server has problems, the whole team can be affected.

6. Distributed Version Control Systems

A Distributed Version Control System, also called DVCS, works differently from the centralized model.

In a distributed system, every developer gets a complete copy of the project. This includes the files, commits, branches, and full history of changes.

In DVCS, every developer’s local machine acts like a complete repository.

How DVCS Works

  • Developers clone or pull the repository from a remote server.
  • The full project history is copied to the developer’s local machine.
  • Developers can make changes locally.
  • Developers can view history without connecting to the server.
  • Developers can create commits while offline.
  • A connection is mainly needed when pulling new changes or pushing local changes.

Simple Example

Imagine a team using Git. Each developer has a complete copy of the project on their own computer. They can make commits, check previous versions, and inspect history even without internet access.

When they are ready to share their work, they push their changes to a remote repository such as GitHub, GitLab, Bitbucket, or another Git server.

Advantages of DVCS

DVCS is powerful because developers can work independently and offline while still keeping full project history.
  • Offline work: Developers can commit changes and view history without an internet connection.
  • Better speed: Many operations are fast because they happen locally on the developer’s machine.
  • Full history everywhere: Every developer has a complete copy of the repository history.
  • Improved collaboration: Developers can work on separate branches and later merge their changes.
  • Better backup: Since many developers have full copies of the repository, the project history is not stored in only one place.

Disadvantages of DVCS

DVCS can feel harder at first because developers need to understand concepts such as commits, branches, merging, cloning, pulling, and pushing.
  • Harder learning curve: Distributed systems can be more complex for beginners.
  • More concepts to understand: Developers need to learn local repositories, remote repositories, branches, merges, and conflict resolution.
  • Repository size: Since the full history is copied locally, very large repositories can take more disk space.

7. CVCS vs DVCS Comparison

Both centralized and distributed version control systems help developers manage source code changes. However, they differ in how they store history and how developers interact with the repository.

Feature CVCS DVCS
Meaning Centralized Version Control System Distributed Version Control System
Main Repository Stored on one central server Stored on a server and copied fully to each developer’s machine
Project History Usually stored mainly on the central server Stored fully on every developer’s local machine
Local Machine Usually contains only a working copy Contains a full repository with complete history
Internet or Server Connection Needed for many operations Needed mainly for pulling and pushing changes
Offline Work Limited offline support Strong offline support
Speed Can be slower because it depends more on the server Usually faster because many operations are local
Learning Difficulty Usually easier to learn Can be harder at first
Access Control Strong centralized access control Access control depends on the remote hosting system and workflow
Examples Subversion, Perforce Git, Mercurial

8. Important Terms to Remember

  • Repository: A storage location for project files and version history.
  • Working copy: The copy of the project files that a developer edits on their own machine.
  • Commit: A saved record of changes made to the code.
  • Pull: Getting the latest changes from a server or remote repository.
  • Push: Sending your local changes to a server or remote repository.
  • History: The record of all changes made to the project over time.
  • Server: A machine or service that stores the repository for team access.
  • Client: A developer’s local machine that connects to the server.

9. Final Summary

Version control systems are essential tools in software development. They help developers track changes, collaborate with others, manage project history, and recover older versions of code when needed.

Centralized version control systems use one central server that stores the main repository and project history. Developers connect to this server to get code, view history, and submit changes.

Distributed version control systems give each developer a full copy of the repository and its history. This makes it possible to work faster, view history locally, and continue working even without a server connection.

The key difference is this: CVCS keeps the full project history mainly on the central server, while DVCS gives every developer a full copy of the project history.
Well done! You now understand the difference between centralized and distributed version control systems.
