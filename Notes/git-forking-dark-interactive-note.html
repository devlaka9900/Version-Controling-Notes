🍴 Git & GitHub Study Notes

Forking Repositories

Forking is a safe way to copy someone else's repository into your own GitHub account, make changes freely, and propose those changes back using a pull request. Think of it as creating your own workspace beside the original project — not inside it.

Best for Open-source contribution
Main idea Your copy + upstream project
Final step Open a pull request

01 · Concept

What is a fork?

A fork is a new repository created from another repository. On GitHub, that original repository is commonly called the upstream repository. Your fork lives under your account, so you can make changes without directly editing the original project.

Simple meaning: “I want my own copy of this project, so I can experiment and maybe send improvements back later.” ✨

🧪 Experiment safely

Try changes without breaking or touching the original repository.

🛠 Fix a bug

Make a fix in your fork, then send it to the project owner through a pull request.

🌱 Start from a project

Use an existing project as a starting point for your own idea, while keeping a link to the original.

02 · Picture it

The fork mental model

The easiest way to understand forking is to separate the project into three places: the original project, your GitHub copy, and your local machine.

Upstream

Original repository

Example: ORIGINAL-OWNER/project

You may not have permission to push here directly.

Fork on GitHub
Clone to computer

Origin + local

Your fork and local copy

Example: YOUR-USERNAME/project

You can push here because it belongs to you.

Important: A fork is not the same as a branch. A branch is inside one repository. A fork is a separate repository that can contain its own branches, issues, pull requests, and settings.

03 · Compare

Fork vs clone vs branch

These three words are related, but they do different jobs. This table keeps them clear.

Term Where it lives What it means Common command or action
Fork GitHub account or organization A separate copy of another repository under your control. Click Fork on GitHub or use gh repo fork.
Clone Your computer A local copy of a remote repository. git clone URL
Branch Inside a repository A separate line of development for a feature, bug fix, or experiment. git checkout -b feature/name

04 · Flow

The complete forking workflow

This is the normal workflow when you want to contribute to a project where you do not directly push to the original repository.

1

Fork

Create your GitHub copy from the original repository.

2

Clone

Download your fork to your computer.

3

Branch

Create a feature branch for your work.

4

Commit + push

Save changes locally, then push to your fork.

5

Pull request

Ask the upstream project to review and merge.

Memory trick: Fork → Clone → Branch → Commit → Push → Pull Request

05 · Commands

Step-by-step commands

The exact repository names will change, but the structure stays the same. Replace the placeholder names with your real GitHub username and repository details.

After clicking Fork on GitHub 
# Clone your fork from GitHub to your computer.
git clone https://github.com/YOUR-USERNAME/REPOSITORY-NAME.git

# Move into the cloned project folder.
cd REPOSITORY-NAME

# Check which remote repositories your local project knows about.
git remote -v
Add the original repository as upstream 
# Add the original project as a remote named upstream.
git remote add upstream https://github.com/ORIGINAL-OWNER/REPOSITORY-NAME.git

# Confirm that origin and upstream are both configured.
git remote -v
Create a branch, commit, and push 
# Create and switch to a new branch for your change.
git checkout -b feature/my-change

# Check which files changed after editing.
git status

# Stage the changed files.
git add .

# Commit the staged files with a clear message.
git commit -m "Add my change"

# Push your branch to your fork on GitHub.
git push -u origin feature/my-change

Then open GitHub: Go to your fork, click Compare & pull request, confirm the base repository is the original project, and submit the pull request.

06 · Remotes

origin vs upstream

In a fork workflow, you usually work with two remote repositories. This is where beginners often get confused, so keep this part clear.

📦 origin

This normally points to your fork.

You push your feature branches here because you own this repository.

Example
origin  https://github.com/YOUR-USERNAME/project.git

🏠 upstream

This points to the original repository.

You fetch new changes from here to keep your fork updated.

Example
upstream  https://github.com/ORIGINAL-OWNER/project.git
Need Remote Example
Push your branch origin git push origin feature/my-change
Fetch updates from original project upstream git fetch upstream
Merge latest original changes into your local main upstream/main git merge upstream/main

07 · Keep updated

How to sync your fork

After you fork a repository, the original project may continue to receive new commits. Your fork does not magically stay updated forever, so you should sync it before creating or updating a pull request.

Sync using normal Git commands 
# Move to your local main branch.
git checkout main

# Download new commits from the original repository.
git fetch upstream

# Merge the original project's main branch into your local main branch.
git merge upstream/main

# Push your updated local main branch to your fork on GitHub.
git push origin main

Sync using GitHub website

  1. Open your fork on GitHub.
  2. Find the Sync fork option near the file list.
  3. Review the incoming commits.
  4. Click Update branch.

This is beginner-friendly, but command line syncing helps you understand what Git is actually doing.

Sync using GitHub CLI 
# Sync your fork's main branch from its parent repository.
gh repo sync YOUR-USERNAME/REPOSITORY-NAME -b main

Conflict reminder: If your fork and the original project changed the same lines differently, Git may ask you to resolve conflicts before the sync can complete.

11 · Avoid mistakes

      
      
Common forking mistakes

      
      

        
        

          
❌ Pushing to the wrong remote

          
In a fork workflow, you usually push your branch to origin, not upstream.

          
Fix: Run git remote -v before pushing.

        

        
        

          
❌ Working directly on main

          
Making feature commits directly on main can make pull requests messy.

          
Fix: Create a feature branch first.

        

        
        

          
⚠️ Forgetting to sync

          
Your fork can fall behind the original project.

          
Fix: Run git fetch upstream and merge before opening a PR.

        

        
        

          
⚠️ Unclear commit messages

          
Messages like “update” or “fix” are hard to review.

          
Fix: Use messages like Fix typo in installation guide.

13 · Save this

      
      
Forking cheat sheet

      
      

        

          Complete command recap
          
        

        
# 1. Clone your fork.
git clone https://github.com/YOUR-USERNAME/REPOSITORY-NAME.git

# 2. Enter the project folder.
cd REPOSITORY-NAME

# 3. Add original repository as upstream.
git remote add upstream https://github.com/ORIGINAL-OWNER/REPOSITORY-NAME.git

# 4. Verify origin and upstream.
git remote -v

# 5. Create a feature branch.
git checkout -b feature/my-change

# 6. Stage changed files.
git add .

# 7. Commit changes.
git commit -m "Describe your change"

# 8. Push feature branch to your fork.
git push -u origin feature/my-change

# 9. Sync your fork later.
git checkout main
git fetch upstream
git merge upstream/main
git push origin main

      

      
      

        
Best beginner habit: Before every contribution, run git remote -v, confirm where origin and upstream point, then create a clean feature branch.
