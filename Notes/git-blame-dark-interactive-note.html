Git Blame: find who changed what, when, and why

git blame shows the commit, author, timestamp, and line number for every line in a file. It is useful when you need to understand the history behind code, investigate bugs, or ask the right teammate for context.

1. What is git blame?

In a team project, many people edit the same files over time. When something looks confusing, you may want to know: who last changed this line, when did they change it, and which commit made that change?

That is where git blame helps. It prints the file line by line and attaches history information to each line.

What file?

You run blame against one specific file, such as feature.js or setup.py.

What line?

Git shows the exact line number and the current content on that line.

What commit?

Git shows the commit ID that last changed that line, plus the author and date.

Important mindset: despite the name, git blame should not be used to attack people. Use it to understand context, trace history, and collaborate better.

2. Basic command

The simplest form is git blame followed by the file name.

Show blame information for one file 
# Show who last changed each line in feature.js
git blame feature.js

Example with a Python file:

Example using setup.py 
# Show blame information for setup.py
git blame setup.py
Simple idea: git log tells you the commit history of the repository. git blame tells you the commit history of each line inside one file.

4. Show only a specific line range

Large files can have hundreds or thousands of lines. You usually do not need the entire file. Use uppercase -L to limit the output to a start line and end line.

Show lines 5 to 15 only 
# Show blame output only from line 5 to line 15
git blame -L 5,15 setup.py
Careful: -L is uppercase for line ranges. Lowercase -l has a different meaning: it shows full-length commit hashes.
Why does Git show the same commit ID on multiple lines?
If one commit changed several lines, the same commit ID can appear on many output lines. That means those lines were last changed together in the same commit.

5. Change the output format

git blame has useful options for making the output more detailed or easier to read.

Show full commit hash

Use lowercase -l when you want the complete commit ID.

Full commit hashes 
# Show full-length commit hashes
git blame -l setup.py

Show author email

Use -e when you want email addresses in the output.

Author emails 
# Show author email addresses
git blame -e setup.py

Change date format

Use --date=short to show simpler dates.

Short dates 
# Show dates in YYYY-MM-DD format
git blame --date=short setup.py

Ignore whitespace changes

Use -w when formatting changes are hiding the real logic change.

Ignore whitespace 
# Ignore whitespace-only differences while blaming
git blame -w setup.py

6. Go from a blamed line to the full commit

git blame shows where a line last changed. To see the actual full commit details, copy the commit ID and inspect it with git show or git log -p.

Option A: use git show

Inspect one commit 
# Show the commit message and patch for this commit
git show 8b55a9c3

Option B: use git log -p

Inspect one commit with patch 
# Show one commit and its detailed patch
git log -p -1 8b55a9c3
Good workflow: use git blame to find the commit ID, then use git show or git log -p -1 to understand the full change.

9. Common mistakes and helpful tips

Using lowercase -l for line ranges
Use uppercase -L for line ranges, like git blame -L 5,15 setup.py. Lowercase -l means show the full commit hash.
Expecting git blame to show uncommitted changes
git blame mainly works with committed history. If you edit a file but do not commit it, that new edit is not part of the commit history yet.
Assuming the blamed author is always the original author
Blame shows who last changed the line. If someone only reformatted code, they might appear as the last author. Use -w to reduce noise from whitespace-only changes.
Forgetting to inspect the full commit
Blame gives a clue. The full story is usually in the commit message and patch. Use git show <commit-id> after finding the commit ID.

Final mental model

Think of git blame as a history label attached to every line of a file. It answers: Which commit last touched this line?

For deeper investigation, combine it with git show or git log -p. Blame points you to the right commit; the commit details explain the actual change.
